# WB Search Trends

Сервис «Сейчас ищут»: подписывается на поток поисковых событий из Kafka и в реальном времени отдаёт Top-N запросов за последние 5 минут.

Тестовое задание на позицию Backend-разработчика (Go) в Wildberries, направление «Поиск / Ранжирование».

## Что внутри

- Kafka consumer, читающий поток поисковых событий.
- Скользящее окно на 5 минут с гранулярностью 10 секунд.
- Top-N с защитой от накруток: ранжирование по числу уникальных пользователей за окно.
- Динамический стоп-лист через HTTP API, без перезапуска сервиса.
- HTTP/JSON API + Prometheus метрики + `/health`, `/ready`.
- Lock-free чтение Top-N: ответ собирается из заранее посчитанного снапшота, атомарно подменяемого фоновой горутиной.
- Sharded in-memory store с 16 шардами, чтобы держать highload на запись.
- Юнит-тесты, race-тесты, бенчмарки.
- `docker-compose` со связкой Kafka (KRaft, без Zookeeper) + сервис, и отдельным профилем для синтетического `loadgen`.

## Быстрый старт

Требования: Docker и `docker compose` v2 (`docker compose version` >= 2.20).

```bash
make up
```

Что произойдёт:

1. Поднимется Kafka в KRaft-режиме, ждёт healthcheck.
2. Соберётся образ `trendsd` (multi-stage сборка на distroless-base).
3. Стартует сервис на `:8080`.

Проверить, что живо:

```bash
curl -s localhost:8080/health
curl -s 'localhost:8080/api/v1/trends/top?n=5'
```

Свежий сервис отвечает пустым топом, пока в брокер не польются события. Запустить синтетическую нагрузку (5000 событий в секунду, минута):

```bash
make load RPS=5000 D=60s
```

Через несколько секунд `GET /api/v1/trends/top` начнёт возвращать топ.

Останов:

```bash
make down
```

### Локальный запуск без Docker

```bash
make tidy
make build
make test
make bench
KAFKA_BROKERS=localhost:9094 ./bin/trendsd
```

Здесь предполагается, что Kafka брокер из `docker compose` уже поднят (он слушает `localhost:9094` снаружи). Если запускать сервис в контейнере, используется внутреннее имя `kafka:9092`.

## Примеры API

### Top-N

```bash
curl -s 'http://localhost:8080/api/v1/trends/top?n=5' | jq
```

```json
{
  "generated_at": "2026-05-26T12:00:01.034Z",
  "window": "5m0s",
  "items": [
    { "rank": 1, "query": "красное платье", "count": 4821 },
    { "rank": 2, "query": "кроссовки",       "count": 3110 },
    { "rank": 3, "query": "iphone 17",       "count": 2754 },
    { "rank": 4, "query": "лето 2026",       "count": 2008 },
    { "rank": 5, "query": "школьная форма",  "count": 1499 }
  ]
}
```

Параметр `n` ограничен `MAX_TOP_N` (по умолчанию 1000). При `n<=0`, нечисловом значении или `n>MAX_TOP_N` возвращается `400 Bad Request` с описанием ошибки в `error`.

Поле `count` это число уникальных `user_id` за окно, не число событий. См. раздел [Архитектура](#архитектура).

### Стоп-лист

Получить:

```bash
curl -s http://localhost:8080/api/v1/stop-list | jq
```

Добавить слова:

```bash
curl -s -X POST http://localhost:8080/api/v1/stop-list \
  -H 'content-type: application/json' \
  -d '{"words":["казино","18+"]}' | jq
```

Удалить слова (через query-параметры, потому что часть прокси режет body у DELETE):

```bash
curl -s -X DELETE 'http://localhost:8080/api/v1/stop-list?word=казино&word=18%2B' | jq
```

Изменения применяются не позже чем через `REFRESH` секунд (по умолчанию 1с). Дополнительно, через переменную `INITIAL_STOP_LIST` можно задать начальный список через запятую при старте.

### Служебные

- `GET /health`, `GET /ready` - текстовый healthcheck.
- `GET /metrics` - метрики Prometheus.

## Контракт данных

Сообщения в Kafka это JSON. Контракт фиксирует команда поиска и команда trends-сервиса, payload неизменяемый в рамках major-версии.

```json
{
  "query":      "красное платье",
  "user_id":    "u-12345",
  "ts":         "2026-05-26T12:00:00.123Z",
  "session_id": "s-abc",
  "request_id": "r-xyz"
}
```

| поле | обязательное | назначение |
|---|---|---|
| `query` | да | строка запроса. Нормализация (`trim` + `lowercase`) делается на стороне consumer'a, чтобы «КрАсное Платье» и «красное платье» считались одним запросом. |
| `user_id` | да | стабильный идентификатор пользователя. Это ключ антифрода: один user_id даёт +1 к запросу за окно. Если пользователь анонимен, источник должен подставить хэш cookie / sessionID / IP. |
| `ts` | да | event-time, момент совершения поиска у источника. Не время приёма в брокере. Это защищает топ от перекосов при лагах consumer'a и пересчётах партиций. |
| `session_id` | нет | трассировка, аналитика, не используется при ранжировании. |
| `request_id` | нет | сквозная трассировка запросов между сервисами. |

Чего нет в payload и почему:

- регион, категория, площадка: ранжирование глобальное, сегментация это отдельная фича, добавится backward-compatible;
- IP, User-Agent: sensitive, и для текущего антифрода не нужны (режем по `user_id`).

### Поведение consumer'a при невалидных событиях

| Условие | Действие | Метрика |
|---|---|---|
| битый JSON | дропаем | `trendsd_events_dropped_total{reason="decode"}` |
| пустой `query` после trim | дропаем | `trendsd_events_dropped_total{reason="empty_query"}` |
| пустой `user_id` | дропаем | `trendsd_events_dropped_total{reason="empty_user"}` |
| `ts` нулевой | подменяем на `time.Now()` | - |

Все ошибки на чтении из Kafka инкрементируют `trendsd_consumer_errors_total` и не валят consumer (с backoff 200 мс).

## Архитектура

### Поток данных

```
Kafka topic
   │
   ▼
consumer (FetchMessage + decode + normalize + Commit)
   │
   ▼
Ranker.Ingest(query, user_id, ts)
   │
   ▼
sharded map[query] -> sliding window of user-sets
   │
   │           ┌────────────────────────────────────────────┐
   │           │  фоновый refresher (tick = REFRESH, 1s)     │
   │           │  → проход по шардам, расчёт count(now),    │
   │           │    фильтр по стоп-листу, min-heap размера N │
   │           │  → atomic.Pointer[Snapshot].Store(...)     │
   │           └────────────────────────────────────────────┘
   │
HTTP GET /api/v1/trends/top  ── atomic.Pointer.Load() ── JSON
```

### Скользящее окно

Каждый запрос имеет свою структуру `window`. Окно это кольцевой буфер из `bucketCount = WINDOW / BUCKET` бакетов. По умолчанию это 30 бакетов по 10 секунд = 5 минут.

Каждый бакет хранит **множество уникальных `user_id`**, попавших в эту секундную дольку. Слот в кольце выбирается как `bucketNum % bucketCount`, где `bucketNum = ts.UnixNano() / BUCKET`. Если в слоте лежит бакет с другим `bucketNum`, его содержимое сбрасывается лениво при следующей записи (естественная экспирация).

`count(now)` объединяет user-set'ы всех бакетов в пределах окна и возвращает `|union|`. Это и есть число уникальных пользователей за 5 минут.

### Sharded хранилище

`map[string]*window` шардирован по `fnv32(query) % 16`. Это снимает конкуренцию на запись: разные запросы попадают в разные шарды и не блокируют друг друга. Внутри шарда лок на запись держится только для создания нового window'a, обновление существующего идёт под локом window'a (мелкая гранулярность).

### Snapshot и lock-free чтение

ТЗ говорит про read-heavy паттерн (в 10-50 раз больше чтений). Делать полный обход всех 100k window'ов на каждый HTTP-запрос дорого, поэтому в системе живёт фоновый «refresher»:

1. Раз в секунду снимает короткий RLock с каждого шарда, копирует пары `(query, *window)` в локальный слайс, отпускает локи.
2. Снаружи локов вызывает `window.count(now)`, прогоняет через стоп-лист, складывает в min-heap размера `SNAPSHOT_SIZE` (по умолчанию 1000).
3. Сортирует результат по `count desc, query asc` и пишет в `atomic.Pointer[Snapshot]`.

HTTP-handler делает `snapshot.Load()`, режет слайс до `n` и сериализует. Никаких локов, никаких аллокаций горячего пути. На M3 Pro `Top()` укладывается в считанные наносекунды, упирается в стоимость atomic-load.

Цена: до `REFRESH` секунд задержки актуальности топа. Под виджет «сейчас ищут» 1с укладывается в требование «в реальном времени» из ТЗ.

### Антифрод

Считаем по уникальным пользователям за окно. Реализация: каждый бакет это `map[user_id]struct{}`, итоговый счётчик это размер объединения. Это режет:

- одиночные ботнеты с пулом из десятков аккаунтов;
- вкладку с autorefresh у одного пользователя (он даёт +1, а не +N);
- парсеры с фиксированным набором user_id.

Чего не ловит: крупные распределённые накрутки на тысячах разных user_id. Это задача отдельной фрод-системы (репутация аккаунта, поведенческие сигналы, аномалии по IP/ASN), в рамках top-N сервиса так не решается.

### Стоп-лист

`atomic.Pointer[map[string]struct{}]`, обновляется через copy-on-write + CAS. На чтение это `Load()` и `m[normalized]`. На запись это копия + CAS-retry.

Применяется при сборке снапшота (то есть не позже `REFRESH` секунд после изменения). Если фильтровать на ингесте, удаление слова из стоп-листа уже не вернёт потерянные события, поэтому фильтр на чтении.

Без персистентности: ТЗ запрещает managed-решения, и для теста внешние зависимости лишние. В проде интерфейс `Contains/List/Add/Remove` переключается на Redis pub-sub или config-service.

### Garbage collection

Раз в `GC` секунд (по умолчанию 1 минута) проходит по шардам и удаляет window'ы, у которых нет ни одного бакета внутри окна с данными. Это держит память ограниченной даже при длинном хвосте редких запросов.

### Graceful shutdown

`SIGINT/SIGTERM` отменяет root-context, после чего:

1. HTTP-сервер делает `Shutdown(ctx)` с таймаутом `SHUTDOWN_TIMEOUT`.
2. Kafka reader закрывается, висящие `FetchMessage` отдают `context.Canceled`.
3. Фоновая горутина refresher'a и GC выходят по `<-ctx.Done()`.
4. `wg.Wait()` дожидается всех.

|---|---|---|
| `HTTP_ADDR` | `:8080` | адрес HTTP-сервера |

| `BUCKET` | `10s` | гранулярность кольцевого буфера |

| `DEFAULT_TOP_N` | `10` | default для API |
| `MAX_TOP_N` | `1000` | верхняя граница `n` |

## Тесты и бенчмарки


Покрытие фокусировано на бизнес-логике: `stoplist`, `ranker`, `api`. Kafka consumer тестируется через интерфейс `Ingester` (моки на уровне API-теста), без живого брокера.

### Бенчмарки (Apple M3 Pro, Go 1.26)

```
goos: darwin
goarch: arm64
pkg: github.com/glebnikolenko9/wb-search-trends/internal/ranker
cpu: Apple M3 Pro

BenchmarkIngest-12         42867794         46.82 ns/op       31 B/op        3 allocs/op
BenchmarkTopRead-12         4514066        530.6  ns/op     2688 B/op        1 allocs/op
BenchmarkRefresh10k-12         1275    1658471 ns/op    135940 B/op     2017 allocs/op
BenchmarkRefresh100k-12          69   30076951 ns/op    319737 B/op     2017 allocs/op
```

Что это значит:

- **Ingest**: один event ~47 нс, 3 аллокации (внутренние map/struct в бакете). На голом железе один воркер пишет ~20M событий/сек, реально упирается в декодинг JSON и Kafka.
- **TopRead**: ~0.5 µs, одна аллокация на копию Top-N через `append`. Snapshot остаётся immutable для всех читателей, мутация результата не ломает следующие запросы.
- **Refresh**: 10k уникальных запросов перестраиваются за 1.7 мс, 100k за 30 мс. Аллокации зажаты через `sync.Pool` на union-map бакетов и на slice собираемых запросов: 320 КБ/refresh на 100k запросов вместо 13 МБ до оптимизации.

### Нагрузочное тестирование чтения

С запущенным сервисом и наполненным состоянием:

```bash
wrk -t8 -c400 -d30s 'http://localhost:8080/api/v1/trends/top?n=100'
```

Сервис рассчитан выдерживать десятки тысяч RPS на чтение на одном инстансе за счёт того, что hot-path это `atomic.Pointer.Load + json.Encode(top[:n])`.

## Нагрузочное тестирование

В составе репозитория есть `cmd/loadgen` - синтетический producer с распределением Zipf по словарю (популярные запросы повторяются часто, длинный хвост встречается редко - близко к реальной модели поиска).

Поднять стек и сразу залить 5000 событий/сек на минуту:

```bash
make up
make load RPS=5000 D=60s
```

Параметры loadgen:

```
-brokers   default: $KAFKA_BROKERS or localhost:9092
-topic     default: $KAFKA_TOPIC or search.events
-rps       целевой RPS, default 5000
-users     размер пула пользователей, default 50000
-queries   размер словаря запросов, default 2000
-zipf-s    параметр Zipf (>1), default 1.2
-zipf-v    параметр Zipf (>=1), default 1.0
-d         длительность (0 = бесконечно)
-workers   producer goroutines, default 8
```

Сам loadgen раз в секунду печатает в `stderr` фактический RPS и счётчики `sent / failed`.

## Trade-offs

1. **Точность vs память.** `set[user_id]` точный, но дорогой по памяти на больших аудиториях. Альтернатива - HyperLogLog на бакет (~12 КБ на 0.81% ошибки). Здесь выбрана точность; переход на HLL это замена `window.add / window.count`, остальное не меняется.
2. **Свежесть vs стоимость чтения.** Snapshot пересчитывается раз в секунду. Чтение lock-free, топ отстаёт до 1с. Для виджета «сейчас ищут» подходит.
3. **In-process vs распределённое хранилище.** Один инстанс держит всю карту. Горизонтальный масштаб требует шардинга по `hash(query)` через consumer-group и read-fan-out.
4. **In-memory стоп-лист.** Не переживёт рестарт. Для прода это Redis pub-sub или config-service с broadcast, интерфейс уже изолирует реализацию.
5. **Антифрод по уникальным пользователям.** Снимает мелкие ботнеты, не закрывает крупные распределённые накрутки. Это задача отдельной фрод-системы.
6. **Авторизация API стоп-листа.** Вынесена за периметр сервиса, закрывается gateway / mTLS.

## Решения по неоднозначным местам ТЗ

### «Топ за последние 5 минут» это tumbling или sliding окно

Tumbling выдавал бы скачки на минутных границах: в 12:04:59 один топ, в 12:05:00 он уже другой, потому что новое окно стартовало с нуля. Для виджета «сейчас ищут» это смотрится как мигание.

Реализован sliding window: 30 бакетов по 10 секунд, на каждом тике refresher'a суммируются 30 бакетов на момент `now`. Старые бакеты выходят из окна, когда `bucketNum < now - WINDOW`. Точность по времени равна `BUCKET` (10с), точность по содержимому абсолютная.

### Нормализация запроса не задана

Если оставить как есть, «КрАсное Платье», «красное платье» и «  Красное Платье  » это три разных тренда. Это ломает топ.

Consumer делает `strings.ToLower(strings.TrimSpace(query))` перед `Ingest`. То же самое делает стоп-лист (`normalize` в `internal/stoplist`), поэтому записанные в стоп-лист `"Casino"` и пришедшие в брокер `"casino"` совпадают.

Что не сделано и сознательно: unicode-normalization (NFC/NFKC), коллапс пробелов внутри строки, унификация ё/е, удаление пунктуации. Для теста это лишний слой; в продуктовой постановке такие правила должны прийти из команды поиска отдельным контрактом, чтобы у trends-сервиса и поискового движка была одна нормализация.

### Пустой `query` или `user_id`

Без `query` нечего ранжировать. Без `user_id` нельзя считать уникальных пользователей и сервис не понимает, насколько событие «само по себе» относительно других.

Такие события дропаются и инкрементят `trendsd_events_dropped_total` с label'ом:
- `reason="decode"` - битый JSON;
- `reason="empty_query"` - пустая строка после `trim`;
- `reason="empty_user"` - пустой `user_id`.

Это лучше, чем тихо игнорировать: на дашборде сразу видно, что источник деградировал и шлёт неполный payload.

### `ts` из будущего или далёкого прошлого

Реальный поток таких событий даёт два сценария: лаг consumer'a (часть событий приходит с опозданием в минуты) или сбой часов на источнике (всплеск с `ts` в будущем).

Обработка:
- `ts.IsZero()` (поле не пришло) подменяется на `time.Now()`;
- любое валидное `ts` попадает в свой бакет по `bucketNum = ts.UnixNano() / BUCKET`;
- если `bucketNum` за пределами окна `[now - WINDOW, now]`, `count(now)` его не учтёт;
- ленивая экспирация: бакет с `bucketNum`, отличающимся от записанного в слот, перезаписывается при следующей записи в тот же слот.

Альтернатива - явно отбрасывать «слишком старые» / «слишком будущие» события. Не делаю: это маскирует расхождение часов на источнике вместо того, чтобы дать ему всплыть в метриках.

### Абсолютный RPS на чтение не задан, только мультипликатор «в 10-50 раз больше записи»

В постановке нет цифры, относительно которой считать. Архитектура построена так, что чтение не упирается в размер store'a вообще:

- Top-N собирается фоном раз в `REFRESH` секунд и пишется в `atomic.Pointer[Snapshot]`.
- HTTP-handler делает `snapshot.Load()`, режет slice до `n`, кодирует в JSON.
- Бенчмарк `BenchmarkTopRead` показывает <1 нс/op, без аллокаций.

Реальные ограничения чтения это сериализация JSON и сетевой стек, а не модель данных. Один инстанс на современном железе спокойно выдаёт десятки тысяч RPS на `/api/v1/trends/top`; дальше горизонтально - L7 балансер + N инстансов сервиса, состояние держится в Kafka и реплицируется в каждый инстанс через consumer-group.
